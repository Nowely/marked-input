## Проблема: Перекрывающиеся сегменты разных паттернов

### Пример 1: Разные паттерны с общими сегментами

```
Паттерны: @[__value__], #[__value__]
Входная строка: "@[hello #[world]]"

Найденные сегменты:
  @[ at 0  ← для @[...]
  #[ at 8  ← для #[...]
  ]  at 15 ← для #[...]? или для @[...]?
  ]  at 16 ← для @[...]? или для #[...]?
```

**Проблема**: Когда ищем закрывающий `]` для внешнего `@[...`, находим первый `]` (позиция 15), который на самом деле закрывает `#[...]`.

### Пример 2: Nested __nested__ (работает ✅)

```
Паттерн: @[__nested__]
Входная строка: "@[hello @[world]]"

Решение: Балансировка по количеству открывающих/закрывающих сегментов
- Если hasNested = true → балансируем
- Считаем @[ и ] между текущей позицией и кандидатом
- Если nestLevel > 0 → пропускаем этот ]
```

**Работает**, потому что все сегменты принадлежат ОДНОМУ паттерну.

### Пример 3: Mixed patterns без __nested__ (НЕ работает ❌)

```
Паттерны: @[__value__], #[__value__]
Входная строка: "@[hello #[world]]"

Проблема: Не можем отличить ] от #[...] и ] от @[...]
- Оба паттерна не имеют __nested__
- Оба имеют сегмент ]
- Нужно понимать контекст: какой ] к какому паттерну относится
```

## Оставшиеся падающие тесты

1. **handles mixed markup types with nesting** - разные паттерны вложены друг в друга
2. **handles HTML-like pattern with label+value containing nested pattern with label only** - HTML + другая разметка
3. **handles complex HTML-like nested structure** - сложная HTML структура
4. **does NOT find nested marks in __label__ sections** - `@[hello #[world]]` с разными паттернами
5. **correctly distinguishes between __label__ and __nested__ in mixed patterns** - микс __value__ и __nested__
6. **handles marks with empty labels** - edge case с пустыми значениями
7. **handles conflicting patterns gracefully** - конфликтующие паттерны

## Возможные решения

### Решение 1: Context-Aware Segment Matching (сложно)

**Идея**: При поиске закрывающего сегмента учитывать контекст - какие паттерны могут быть между открывающим и закрывающим сегментами.

**Алгоритм**:
1. Начинаем с открывающего сегмента паттерна A
2. При поиске закрывающего сегмента:
   - Анализируем все промежуточные сегменты
   - Определяем, какие паттерны могут там быть
   - Считаем "вес" каждого закрывающего сегмента
   - Выбираем правильный

**Сложность**: 
- **Когнитивная**: Высокая (8/10)
- **Код**: ~150-200 строк дополнительной логики
- **Время**: 1-2 дня разработки + тестирование

**Риски**:
- Может ломать существующую логику
- Сложно отладить
- Производительность может упасть

### Решение 2: Greedy First, Balance Later (средне)

**Идея**: 
1. Для паттернов БЕЗ `__nested__`: берем первый matching сегмент (greedy)
2. Для паттернов С `__nested__`: балансируем
3. После нахождения всех матчей - фильтруем конфликты

**Алгоритм**:
```typescript
if (!descriptor.hasNested) {
  // Greedy - take first matching segment
  take_first_match()
} else {
  // Balance nesting
  balance_brackets()
}

// Then filter overlaps
filter_conflicts()
```

**Проблема**: Greedy подход все равно берет неправильный `]` в случае `@[hello #[world]]`.

**Сложность**:
- **Когнитивная**: Средняя (5/10)
- **Код**: ~50-100 строк
- **Время**: 0.5-1 день

**Риски**:
- Может не решить все случаи

### Решение 3: Two-Pass Matching (проще, но медленнее)

**Идея**: 
1. **Pass 1**: Найти все inner-most паттерны (без вложенностей)
2. **Pass 2**: Найти внешние паттерны, исключая уже занятые позиции

**Алгоритм**:
```
Pass 1: Find all patterns that don't contain other patterns
  → #[world] found at [8, 16]
  
Pass 2: Find patterns that may contain Pass 1 results
  → @[...] должен искать ] ПОСЛЕ позиции 16
```

**Сложность**:
- **Когнитивная**: Низкая (3/10)
- **Код**: ~80-120 строк
- **Время**: 0.5-1 день
- **Производительность**: Хуже (двойной проход)

**Риски**:
- Снижение производительности
- Может не работать для сложных nested случаев

### Решение 4: Segment Ownership Tracking (сложно, но правильно)

**Идея**: Трек ownership сегментов - какой сегмент к какому паттерну относится.

**Алгоритм**:
1. При нахождении матча, помечаем все его сегменты как "owned"
2. При поиске нового матча, пропускаем "owned" сегменты
3. Обрабатываем в порядке приоритета (longest first, most specific)

**Сложность**:
- **Когнитивная**: Высокая (7/10)
- **Код**: ~120-150 строк
- **Время**: 1-1.5 дня

**Преимущества**:
- Правильное решение
- Покроет все edge cases
- Понятная логика
