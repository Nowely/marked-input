# Исследование AC-паттерн матчинга - Итоговый отчет

## Цель исследования

Проверить жизнеспособность трех подходов к оптимизации PatternProcessor в ParserV2 с помощью модифицированных алгоритмов Aho-Corasick.

## Реализованные подходы

### 1. Pattern-Level Aho-Corasick (AC1)
**Концепция**: Построить AC-автомат для поиска последовательностей сегментов

**Результат**: ❌ Не жизнеспособен
- **Производительность**: 0.92x (на 8% медленнее текущей V2)
- **Проблемы**: 
  - Перекрывающиеся сегменты требуют NFA-подобного подхода
  - Множественные активные состояния → большой overhead
  - Не дает прироста производительности

### 2. Two-Level Aho-Corasick (AC2)  
**Концепция**: Двухуровневая иерархия - AC для сегментов + AC для паттернов

**Результат**: ✅ Частично жизнеспособен
- **Производительность**: 1.38x (на 38% быстрее в среднем!)
  - 10 марок: 0.96x (чуть медленнее)
  - 50 марок: **1.56x быстрее** ✓
  - 100 марок: **1.63x быстрее** ✓
- **Проблемы**:
  - Все еще требует ручной обработки перекрывающихся сегментов
  - 20 тестов на nested marks не проходят (из 102)
  - Требует доработки логики nested content

### 3. Extended AC with Gaps (AC3)
**Концепция**: Модифицированный AC с поддержкой gap-переходов

**Результат**: ❌ Не жизнеспособен
- **Производительность**: 0.43x (на 57% медленнее!)
- **Проблемы**:
  - Взрыв количества состояний
  - Нарушает гарантии сложности O(n)
  - Очень сложная реализация
  - Худший результат из трех

## Ключевая проблема: Перекрывающиеся сегменты

Для паттерна `@[__value__](__meta__)`:
```
Входная строка: "@[hello](world)"
Найденные сегменты: [@[, ], ](, )]
                         ↑   ↑
                    перекрываются!
```

AC находит оба сегмента `]` и `](`, которые начинаются в одной позиции. Это требует:
- Либо NFA-подхода с множественными состояниями (AC1)
- Либо пост-обработки для выбора правильной последовательности (AC2)
- Либо очень сложной логики gap-переходов (AC3)

## Выводы

### 1. AC2 показывает обещающие результаты
- **+38% производительности** на реальных данных (50-100 марок)
- Хорошая масштабируемость
- Чистая архитектура

### 2. Критическое препятствие
Перекрывающиеся сегменты - фундаментальная проблема, которая:
- Ломает классический AC
- Требует значительной доработки
- Влияет на обработку nested content

### 3. AC1 и AC3 не подходят
- AC1: производительность хуже + сложная реализация
- AC3: значительно хуже производительность + чрезмерная сложность

**Усилия**: Большие (3-5 дней)

3. Добавить автоматический выбор стратегии

## Столкнувшиеся проблемы

1. **Перекрывающиеся сегменты** - основная техническая сложность
2. **Nested content** - требует специальной обработки в AC2
3. **Тестирование** - сложность в обеспечении 100% совместимости
4. **Gap validation** - необходима пост-обработка после AC
