## Проблема: Перекрывающиеся сегменты

### Суть проблемы

Когда используются **разные паттерны** с **общими сегментами**, алгоритм не может определить, какой закрывающий сегмент к какому паттерну относится.

**Пример**:
```
Паттерны: @[__value__], #[__value__]
Вход: "@[hello #[world]]"

Сегменты:
  @[ at 0
  #[ at 8
  ]  at 15  ← для #[...] или @[...]?
  ]  at 16  ← для #[...] или @[...]?
```

Текущий greedy алгоритм берет первый доступный `]` (позиция 15), но для `@[...]` нужен второй (позиция 16).

### Почему не возникает при `__nested__`

Когда паттерн имеет `__nested__`, мы балансируем скобки:
```
Паттерн: @[__nested__]
Вход: "@[hello @[world]]"

Алгоритм:
1. Находим первый ] at 15
2. Считаем: между текущей позицией и 15 есть один незакрытый @[
3. nestLevel = 1 > 0 → пропускаем
4. Находим второй ] at 16
5. nestLevel = 0 → используем
```

Это работает, потому что **все сегменты принадлежат одному паттерну**.

### Почему не работает для mixed patterns

```
Паттерны: @[__value__], #[__value__]
Вход: "@[hello #[world]]"

Проблема:
- @[__value__] НЕ имеет __nested__ → не балансируем
- Берем первый ] at 15 (greedy)
- Но этот ] на самом деле закрывает #[...]
```

## Решение: Segment Ownership Tracking

### Концепция

1. Обрабатываем паттерны по приоритету (longest/most specific first)
2. При нахождении матча, помечаем все его сегменты как "занятые"
3. Последующие паттерны не могут использовать занятые сегменты
4. Это естественным образом решает конфликты

### Логичность механики

**Логична ли текущая проблема?**

**Да**, это фундаментальная проблема любого parser'а:
- Greedy vs non-greedy matching
- Ambiguous grammars
- Conflict resolution

**Примеры из других систем**:
- RegExp: greedy `.*` vs non-greedy `.*?`
- XML/HTML: правильное закрытие тегов
- Скобки в выражениях: балансировка

**Механика логична**, потому что:
1. Для `__nested__` → балансировка естественна
2. Для `__value__` → greedy естественен
3. Для mixed → нужен ownership (стандартное решение)
