# ParserV2 - Правила работы

Лаконичный набор правил, определяющих поведение ParserV2 при парсинге вложенных markup-конструкций.

## 1. Архитектурные принципы

### 1.1 Однопроходной алгоритм
1. **Один проход** для поиска всех совпадений
2. **Один проход** для построения дерева токенов
3. **Нет рекурсивного парсинга** содержимого токенов

### 1.2 Позиционная семантика
1. Все позиции относятся к **оригинальному тексту**
2. `start` - **inclusive** (указывает на первый символ)
3. `end` - **exclusive** (указывает на символ после последнего)
4. Совместимость с `substring(start, end)`

### 1.3 Структура токенов
```typescript
TextToken: { type: 'text', content, position: {start, end} }
MarkToken: { type: 'mark', content, children: [], data: {label, value?, optionIndex}, position: {start, end} }
```

## 2. Markup Pattern Rules

### 2.1 Формат паттернов
1. Паттерн состоит из **статических сегментов** и **плейсхолдеров**
2. Плейсхолдеры: `__label__` и `__value__`
3. Примеры валидных паттернов:
   1. `@[__label__]` - один label
   2. `@[__label__](__value__)` - label и value
   3. `<__label__>__value__</__label__>` - два label (симметричный)

### 2.2 Валидация паттернов
1. `__label__` может встречаться **1 или 2 раза**
2. `__value__` может встречаться **0 или 1 раз**
3. `__value__` **не может** появляться раньше первого `__label__`
4. Паттерн должен содержать **хотя бы один статический сегмент**

### 2.3 Trigger и симметрия
1. **Trigger** = первый символ первого сегмента (используется для группировки)
2. **Symmetric pattern** = первый и последний сегменты одинаковые (`**text**`, `*text*`)

## 3. Алгоритм поиска совпадений

### 3.1 Segment Matching (Aho-Corasick)
1. Все статические сегменты из всех паттернов собираются в единый список
2. Алгоритм **Aho-Corasick** находит все вхождения сегментов в тексте
3. Результат: `UniqueMatch[]` - уникальные сегменты по позиции+значению

**Сложность:** O(N + M), где N = длина текста, M = количество паттернов

### 3.2 Pattern Building (Chain Management)
1. **Pattern Chain** = цепочка сегментов, формирующая один паттерн
2. Цепочка создается при нахождении **первого сегмента** паттерна
3. Цепочка расширяется при нахождении **следующих сегментов**
4. Цепочка завершается при нахождении **последнего сегмента**

### 3.3 Правила приоритетов паттернов

#### При старте новой цепочки:
1. **Более длинные первые сегменты** имеют приоритет (избежание конфликтов `*` vs `**`)
2. При равной длине - **меньше сегментов** = выше приоритет

#### При расширении/завершении цепочки:
1. **Цепочки без вложений** имеют приоритет (закрываются первыми)
2. **Lookahead**: если следующий сегмент доступен сразу - приоритет расширению
3. При прочих равных: **LIFO** - позже начатая = выше приоритет (вложенная)

### 3.4 Pattern Exclusivity
1. Паттерн **не может** начать новое совпадение, пока активна его цепочка
2. Пример: `@[simple]` и `@[simple](value)` не могут быть активны одновременно
3. Короткий паттерн, завершившись, **отменяет** более длинный с той же стартовой позицией

### 3.5 Фильтрация overlaps
После построения всех совпадений удаляются:
1. **Partial matches** - совпадения, являющиеся частью более длинного с тем же началом/концом
2. **Matches inside __value__** - совпадения внутри value-секции другого совпадения

Сохраняются:
1. **Вложенные совпадения** в label-секциях (для построения дерева)

## 4. Построение дерева токенов

### 4.1 Single-Pass Tree Building
Алгоритм обходит отсортированные совпадения **один раз** с использованием стека:

```
for each match in sorted_matches:
  1. Pop completed parents (match не содержится в их label)
  2. Skip если match начинается раньше текущей позиции (конфликт)
  3. Push match в стек для потенциальных children
  
Finalize remaining stack
```

**Сложность:** O(N log N) для сортировки + O(N) для построения

### 4.2 Правила вложенности
1. **Parent-child связь**: child полностью содержится в **labelStart..labelEnd** родителя
2. **Value-секции не парсятся**: совпадения внутри value игнорируются
3. **Self-nesting не поддерживается**: паттерн не может быть вложен сам в себя

### 4.3 Структура children
1. `children` содержит **чередование** TextToken и MarkToken
2. Всегда начинается и заканчивается TextToken (даже пустыми)
3. `children` **пустой массив** если нет вложенных MarkToken (только text)

### 4.4 Позиции текстовых токенов
1. TextToken создается **между** MarkToken
2. Пустые TextToken создаются на границах (совместимость с legacy)
3. Позиции: `{start: prevMarkEnd, end: currentMarkStart}`

## 5. Edge Cases & Guarantees

### 5.1 Обработка граничных случаев
1. ✅ Пустой ввод → `[TextToken("", 0, 0)]`
2. ✅ Текст без markup → `[TextToken(text, 0, length)]`
3. ✅ Пустые label/value → `""` (валидно)
4. ✅ Adjacent marks → корректная позиционная семантика
5. ✅ Unicode/Emoji → полная поддержка
6. ✅ Вложенные скобки в content → `@[text [with] brackets]` работает

### 5.2 Гарантии безопасности
1. 🛡️ **XSS защита** - валидация опасных паттернов
2. 🛡️ **DoS prevention** - защита от бесконечных циклов
3. 🛡️ **Валидация структуры** - проверка корректности дескрипторов
4. 🛡️ **Position validation** - start ≤ end для всех токенов

### 5.3 Не поддерживается
1. ❌ **Self-nesting** - `@[outer @[inner]]` не создаст вложенность для одного паттерна
2. ❌ **Parsing inside __value__** - value рассматривается как plain text
3. ❌ **Bracket counting** - паттерн закрывается первым закрывающим сегментом

## 6. Производительность

### 6.1 Сложность алгоритма
1. **Segment matching:** O(N + M) - Aho-Corasick
2. **Pattern building:** O(Z × S) где Z = matches, S = avg segments per pattern
3. **Tree building:** O(N log N) для сортировки + O(N) для построения
4. **Итого:** O(N log N) вместо O(N × 2^D) у рекурсивного подхода

### 6.2 Benchmark results
1. **35-105x улучшение** для вложенных структур vs рекурсивный парсинг
2. **65,710 ops/sec** для depth=1
3. **47,923 ops/sec** для depth=3

## 7. Примеры

### Простой случай
```typescript
Input:  "Hello @[world](test)"
Markup: "@[__label__](__value__)"
Output: [
  TextToken("Hello ", 0, 6),
  MarkToken("@[world](test)", 6, 20, children=[], data={label:"world", value:"test"}),
  TextToken("", 20, 20)
]
```

### Вложенность
```typescript
Input:  "@[hello #[world]]"
Markups: ["@[__label__]", "#[__label__]"]
Output: [
  TextToken("", 0, 0),
  MarkToken("@[hello #[world]]", 0, 17, children=[
    TextToken("hello ", 2, 8),
    MarkToken("#[world]", 8, 16, children=[], data={label:"world"}),
    TextToken("", 16, 16)
  ], data={label:"hello #[world]"}),
  TextToken("", 17, 17)
]
```

### Конфликтующие паттерны
```typescript
Input:  "@[simple] @[with](value)"
Markups: ["@[__label__]", "@[__label__](__value__)"]
Output: [
  TextToken("", 0, 0),
  MarkToken("@[simple]", 0, 9, ...),  // Короткий паттерн завершился первым
  TextToken(" ", 9, 10),
  MarkToken("@[with]", 10, 17, ...),  // Длинный паттерн отменен, короткий выбран
  TextToken("(value)", 17, 24)
]
```
